# WorldForge 安全策略

## 支持范围

当前仓库处于V1.0开发阶段。安全问题范围包括：

- Electron窗口、Preload与IPC隔离。
- 项目路径、文件导入和导出边界。
- SQLite数据完整性、Migration和备份恢复。
- API凭据处理。
- AI候选越权写入、锁定块破坏和跨项目写入。
- 日志、诊断包和隐私数据泄露。

## 报告安全问题

请通过GitHub仓库的私有安全报告能力提交，不要在公开Issue中粘贴：

- API密钥或访问令牌。
- 未公开作品正文。
- 本地数据库、备份或诊断包。
- 可直接利用的高风险攻击步骤。

报告建议包含：

1. 受影响版本或提交。
2. 操作系统和运行环境。
3. 影响范围。
4. 最小复现步骤。
5. 是否可能造成数据泄露、正文修改或跨项目访问。
6. 可选修复建议。

## 安全基线

WorldForge必须保持：

```ts
nodeIntegration: false
contextIsolation: true
sandbox: true
webSecurity: true
```

同时：

- Renderer不能直接访问Node、文件系统、SQLite、环境变量和凭据。
- Preload只暴露具名白名单方法，不暴露通用IPC发送器。
- 所有IPC和模型结构化输出必须通过Schema校验。
- 项目路径在Core中归一化并限制到允许目录。
- AI输出必须先成为Candidate；未经作者确认不得写入Draft。
- 锁定块由界面和Core双层校验。
- Version不可变。
- 凭据只保存于操作系统凭据服务。
- 默认日志不保存正文、完整Prompt和凭据。

## 不属于漏洞的情况

- 用户主动将项目放入第三方同步盘后产生的同步冲突。
- 用户自行运行或配置的不可信模型服务记录请求内容。
- 用户设备、系统账户或磁盘本身已被完全控制。
- 未验证模型输出质量差，但未突破代码硬约束。

## 修复优先级

| 级别 | 示例 | 目标处理 |
|---|---|---|
| Critical | 未确认AI文本写入正文、跨项目读写、凭据泄露 | 立即阻断发布并优先修复 |
| High | 路径越界、导入处理导致任意文件访问、备份恢复覆盖原项目 | 当前里程碑内修复 |
| Medium | 日志包含过多上下文、部分安全设置可被错误配置 | 排入最近修复版本 |
| Low | 安全提示或防误操作体验不足 | 正常任务处理 |

## 依赖与更新

引入生产依赖前必须记录维护状态、许可证和安全影响。Electron与原生模块升级必须重新运行安全、打包和IPC测试。
