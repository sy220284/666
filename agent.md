# WorldForge Agent 快速执行入口

> `AGENTS.md` 是仓库级完整且唯一的权威指令。本文件只保留高频执行规则；冲突、缺项或细节不明时，立即回到 `AGENTS.md` 和任务卡，不得自行补全规则。

## 1. 必读顺序

```text
AGENTS.md
→ docs/PROJECT_EXECUTION_ENTRY.md
→ docs/tasks/ACTIVE_TASK.json
→ docs/tasks/ACTIVE_TASK.md
→ ACTIVE_TASK 指向的任务卡
→ 任务卡列出的专项文档
→ 真实代码、测试、Migration、契约和最近提交
```

`ACTIVE_TASK.json` 是授权模式、活动任务、分支、允许路径、禁止路径和验证命令的机器真源。不得在本文件中固化当前任务、当前阶段、任务数量或授权模式。

## 2. 最短可靠执行路径

```text
精准识别目标、非目标和适用范围
→ 核对任务、依赖、基线、分支和允许路径
→ 识别上游、下游及横向、纵向关联影响
→ 检查真实实现、测试、Migration、契约和最近变更
→ 建立失败复现或验收测试
→ 完成最小完整端到端实现和必要 UI
→ 验证主路径、边界路径、失败路径及关联回归
→ 复查功能、性能、体验、安全、数据和恢复
→ 同步契约、文档、追踪矩阵、任务状态和 Evidence
→ 永久门禁、Controlled Merge、Main Verification
→ 重新读取 main 后再声明完成
```

重复任务优先复用已经验证的行动路径。失败先定位原因，不得通过绕过门禁、缩小验证范围或伪造成功推进。

## 3. 整体协调与代码质量

所有修复和新增必须从整体配合与协调出发，考虑：

- 横向：相邻功能、共享组件、公共契约、通用状态、共用测试、性能预算和用户体验；
- 纵向：Renderer、Preload、Electron Main、Core、Repository、SQLite、Migration、IPC、Evidence 和发布链路；
- 修改前识别上游输入、下游消费者、数据生命周期、失败传播和恢复路径；
- 修改后执行关联回归，确认没有引入新的功能、数据、安全、性能、兼容性或体验问题；
- 功能实现必须同时满足正确性、性能预算和可用体验。

代码必须同时具备：

```text
规范性
+ 健壮性
+ 可读性
+ 可维护性
+ 可扩展性
+ 可测试性
+ 性能可控性
+ 可观测性
+ 安全性
```

禁止用过度设计替代最小完整实现，也不得以局部优化破坏整体架构、单一真源、恢复能力或后续任务。

## 4. 五项硬边界

1. 项目正文、数据库、设置、索引、日志、Prompt、Eval、配置和备份必须仅保存在用户本机，不得接入云存储或云同步。
2. AI 输出先成为 Candidate，未经作者接受不能进入 Draft。
3. `project.sqlite` 是唯一项目数据真源。
4. Lock、Revision、Hash、不可变 Version、事务、项目与路径边界由代码保证。
5. AI 只能提议，作者拥有最终裁决权。

任一边界失败，立即阻断。

## 5. 分支、实现与验证

- 同时只能有一张活动任务、一个正式任务集成分支、一个 Ready PR 和一个最终受检 Head。
- 辅助分支不得直接向 `main` 开普通任务 PR。
- 两个及以上正式文件必须优先使用 Git Blob/Tree/Commit 形成单个原子提交。
- 不得修改 `ACTIVE_TASK.allowedPaths` 之外的文件，不得重构无关代码。
- 不得用 TODO、空实现、固定成功、硬编码演示数据、吞错、只写测试或临时 Runner 结果冒充完成。
- 修复必须复验原问题；新增必须验证主路径、边界路径、失败路径和用户可操作闭环。
- 必须运行任务卡要求的专项检查；未经真实执行不得声称通过。
- 门禁前后工作树必须干净，关键文件写入后必须从真实 PR Head 重新读取。

## 6. 必须停止并重建的情况

- 写入错误仓库、分支或路径；
- CI 生成的正式源码不在 PR Head；
- 出现任务专属 Workflow、补丁目录或临时脚本压过正式实现；
- 不同门禁验证不同 Tree；
- 任务、Evidence、主线来源或 ACTIVE_TASK 状态不一致；
- 数据库、Migration、安全、项目边界、事务、恢复或关键性能预算失败。

## 7. 完成声明

只有以下条件全部成立，才能声明主线闭环：

- 受检 Head 的永久门禁真实通过；
- Controlled Merge 已实际完成；
- Main Verification 已成功；
- 必要状态已进入 `main`；
- 已重新读取真实 `main` 和关键文件；
- 原始要求已逐项复核；
- 关联功能、性能、体验、安全和数据未出现新回归。

Runner 成功、PR 可合并、Artifact 上传或补丁生成不能单独证明完成。

## 8. 中文提交与协作说明

- Git 提交标题与说明、PR 标题与描述，以及推送、同步、合并、关闭、回滚和诊断中的人工说明必须使用中文。
- `fix`、`feat`、`chore` 等英文类型前缀改为“修复”“功能”“维护”“文档”“测试”“重构”等中文表述。
- 文件路径、分支名、代码标识符、命令和技术专名可以保留英文，但上下文说明必须使用中文。
- Controlled Merge 使用 PR 标题生成 main 的 squash 提交标题，因此 PR 标题必须准确、完整并使用中文。

推荐格式：

```text
修复(M4-01)：修正索引队列重建顺序
功能(M4-01)：实现项目词典管理
文档：优化 Agent 执行与质量规则
```
