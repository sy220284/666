## 3. 已确认问题与整改

### 3.1 M0-03：数据库幂等缓存可淘汰待处理写入

原实现达到1000条上限后直接删除最早Map项，没有区分已完成结果和仍在写队列中的Promise。高并发下，同一`requestId`可在原写入完成前再次进入队列，破坏只执行一次语义。

整改：新增统一的`BoundedIdempotentPromiseCache`。待处理请求不参与淘汰；已完成结果按上限回收；失败请求立即释放以允许显式重试；数据库关闭时清空缓存。

### 3.2 M1-02：项目生命周期幂等缓存存在同类竞态

项目创建、打开、关闭、移动和恢复登记共享生命周期队列，但原缓存达到上限后同样可能删除尚未执行的操作。

整改：项目生命周期改用同一待处理安全缓存，确保排队中的创建、打开、关闭和移动请求不会因容量治理失去幂等保护。

### 3.3 M0-02：凭据核心组件未执行权威Schema校验

IPC层会校验`providerId`和`credentialRef`，但CredentialBroker自身只检查非空或前缀。内部调用和被篡改的凭据文件可以绕过正式合同边界。

整改：Broker写入、读取和公开操作统一使用`ProviderIdSchema`与`CredentialRefSchema`；非法持久化记录按凭据库损坏处理；保留安全存储后端、串行mutation和临时文件清理语义。

### 3.4 M3-07：Renderer请求协调器永久保留正文参数键

协调器使用完整参数序列化字符串作为请求键，并在永久generation Map中保留。草稿Patch、规划内容等正文数据会随唯一请求不断累积，形成无界内存增长和敏感正文长期驻留。

整改：代际号只从当前活动请求派生。请求完成后仅保留零个历史键；并发替换仍使用1→2代际隔离，顺序请求重新从1开始。

### 3.5 M1-01：应用设置并发补丁存在丢失更新

原实现先在写队列外读取设置并合并补丁，再排队写入。多个独立字段并发更新会基于同一旧快照，后写入覆盖先写入。

整改：读取当前设置、恢复无效值、合并补丁和UPSERT全部置于同一串行写事务内，保证不同字段并发更新可累积。

## 4. 回归覆盖

- 幂等缓存超过容量时保留全部待处理请求，只淘汰已完成结果。
- 失败请求从缓存移除，同一`requestId`可重新提交。
- 三个独立应用设置字段并发更新后全部保留。
- Broker拒绝非法providerId和被篡改的凭据文件。
- Renderer顺序请求不保留历史请求键，活动替换代际语义保持不变。
