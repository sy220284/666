## 3. 本轮发现与整改

### 3.1 M1-09：导入预览计划无界保留，提交阶段存在源文件复验缺口

原实现将完整 `ImportPlan` 长期保存在进程内 Map 中，只有成功提交时删除；连续预览可持续占用内存。提交时直接读取预览路径，未先重新检查普通文件、符号链接和 20 MiB 上限，文件在预览后被替换时可能先触发大文件读取。

整改：

- 新增 `CoordinatedImportExportService`，生产入口统一使用协调服务；
- 每个预览计划使用独立基础服务实例，默认最多保留 16 个计划；
- 自动清理超时计划并淘汰最旧计划；
- 提交前重新执行绝对路径、普通文件、符号链接、真实路径和 20 MiB 上限校验；
- 失效计划返回稳定的 `IMPORT_PLAN_STALE`，超限文件返回 `IMPORT_ARCHIVE_LIMIT`；
- 增加计划淘汰和超限替换文件不读取的集成测试。

### 3.2 M1-06：自动保存协调器销毁后仍可能发布迟到状态

编辑器切换章节、离开面板或卸载时会销毁自动保存协调器，但已在执行的保存 Promise 仍可能在完成后发布 `saved` 或 `failed`，覆盖新编辑器状态或触发卸载后的状态更新。

整改：

- 销毁时清除脏状态并保证幂等；
- 保存完成后再次检查销毁状态；
- 销毁后禁止发布任何迟到状态；
- 增加在保存执行期间销毁的单元回归测试。

### 3.3 M0-02：凭据文件并发读改写可能丢失记录

多个凭据保存或删除并发执行时，各操作可能读取同一个旧快照并分别覆盖最终文件，导致已成功返回的凭据引用随后不存在。

整改：

- 所有凭据写操作进入同一串行 mutation 链；
- 查询和解析等待既有 mutation 完成，保证进程内读写一致性；
- 临时文件在成功或失败后统一清理；
- 强化凭据文件结构和记录字段校验；
- 增加 16 个并发保存、解析和删除的单元回归测试。
