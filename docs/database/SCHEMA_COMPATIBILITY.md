# WorldForge Schema兼容策略

> 状态：Approved

## 1. 兼容目标

WorldForge必须在升级应用时安全打开旧项目，并明确处理高版本项目、损坏项目和未完成迁移。兼容策略优先保护正文和历史Version，不追求无限向后兼容。

## 2. 版本字段

- 应用版本：语义版本，如`1.0.0`。
- `app.sqlite` Schema版本：整数递增。
- `project.sqlite` Schema版本：整数递增。
- IPC协议版本：独立整数。
- 导入导出Manifest版本：独立整数。

这些版本不可混用。

## 3. 支持规则

| 场景 | 行为 |
|---|---|
| 数据库版本等于应用支持版本 | 正常读写 |
| 数据库版本低于支持版本 | 创建恢复点后执行Migration |
| 数据库版本高于应用支持版本 | 只读打开并提示使用更新应用，不降级写入 |
| checksum与已应用Migration不一致 | 停止写入，进入诊断与恢复 |
| `quick_check`失败 | 只读打开，允许导出和恢复 |
| 派生索引版本落后 | 正常打开，后台或手动重建索引 |

## 4. V1兼容窗口

V1.x应用至少支持从V1.0发布Schema逐级升级到当前版本。正式V2决策时再确定是否继续支持所有V1早期Schema。

禁止直接跳过中间Migration；升级器按顺序执行。

## 5. 字段兼容

- 新增可空字段：兼容性低风险。
- 新增非空字段：必须提供确定默认值或分阶段回填。
- 枚举新增值：旧应用读取高版本数据库时不得写回未知值。
- 字段改名：先新增新字段并双读迁移，后续版本再移除旧字段。
- 字段语义变化：必须新建字段或表，不复用旧字段名。

## 6. IPC兼容

- Main、Preload、Renderer和Core启动时交换`protocolVersion`。
- 版本不匹配时停止业务命令，仅允许健康检查和安全退出。
- 新增可选字段保持向后兼容；删除或改义字段必须提升协议版本。

## 7. 导出包兼容

`.wfproj` Manifest至少包含：

```json
{
  "formatVersion": 1,
  "projectSchemaVersion": 1,
  "appVersion": "1.0.0",
  "createdAt": "ISO-8601",
  "files": []
}
```

导入高版本格式时只允许预览兼容信息，不尝试猜测降级。

## 8. 派生数据

以下数据可因版本不兼容直接重建：

- FTS5索引。
- 字数与统计缓存。
- 页面布局缓存。
- 约束包缓存。
- AI摘要与日记索引。

重建不得修改正文、Canon、状态、Candidate和Version。

## 9. 兼容测试

- 每个已发布Schema保留最小Fixture。
- 新版本对所有支持Fixture执行升级、核心查询、编辑、Candidate采用、Version创建和导出。
- 高版本Fixture验证只读打开。
- 未知枚举、缺少派生表和部分索引损坏均需测试。

## 10. 退回旧版本

WorldForge不承诺数据库降级。用户需要退回旧应用时，应恢复升级前备份。升级界面必须明确这一点。
