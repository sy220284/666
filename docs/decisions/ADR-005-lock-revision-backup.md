# ADR-005：锁定、Revision、事务与备份共同保护正文

- 状态：Frozen
- 日期：2026-07-13

## 背景

单一保护机制不足以应对编辑并发、AI旧结果、批量替换、结构调整、误删和数据库损坏。需要在编辑、事务和恢复三个层面共同保护。

## 决策

1. 锁定块由Tiptap界面插件和Core LockGuard双层校验。
2. Draft每次提交事务增加一次Revision；AI任务记录基础Revision。
3. 块修改使用`expectedHash`检测目标内容是否已变化。
4. Candidate采用、Version创建、状态确认、拆并章、导入和Migration使用明确事务。
5. 高风险操作前创建重大操作恢复点。
6. 备份分为日常滚动、重大操作和手动命名三轨。
7. 恢复默认生成新项目副本，不直接覆盖原项目。

## 结果

### 正面

- 锁定段落不依赖模型自觉。
- 旧候选不会静默覆盖新编辑。
- 高风险操作和数据库异常均有恢复路径。

### 代价

- 所有写命令需要统一Patch、Revision和事务规范。
- 备份会占用本地空间，需要统计与安全清理。

## 强制约束

- 任何AI、替换、拆章、并章和跨章移动操作都必须经过LockGuard。
- Revision冲突不得自动选择候选文本覆盖当前文本。
- 自动保存不创建不可变Version；Version只在明确动作时创建。
- 最后一份已验证备份不得被自动清理。

## 验证

- 锁定块破坏率为0。
- Revision冲突静默覆盖率为0。
- 重大操作失败后原项目保持可用。
- 从三种备份均可恢复到新目录并通过完整性检查。
