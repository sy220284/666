# ADR-007：主题只影响视觉层，不得分叉业务逻辑

- 状态：Frozen
- 日期：2026-07-14

## 背景

REQ-047要求提供两种视觉方向：Theme A安静编辑部（当前`UI_SYSTEM.md`冻结方案）和Theme B水墨印章（不同排版语言与候选采纳的"盖章"交互隐喻）。M4-05此前已规划浅色/深色/护眼/高对比四种对比模式，但那四种只做重新着色，不改变交互隐喻；Theme B需要更换候选采纳的视觉表现，风险是实现时为两套主题各写一套业务逻辑，导致行为不一致、维护成本翻倍，且违反ADR-004确立的"AI影响必须可追溯到统一的Candidate/StateProposal流程"。

## 决策

1. 主题（含视觉方向和对比模式）只能改变Design Token（颜色、字体、间距、圆角、动效曲线）和组件的视觉表现，不得改变IPC契约、事件协议、状态机或数据模型。
2. `candidate.apply`等底层命令与`ApplyRecord`等数据结构在所有主题下完全一致；Theme B的"盖章"只是该命令触发后的一段动画表现，动画播放失败或跳过不影响底层Patch应用结果。
3. 组件层禁止出现`if (theme === 'B') { ...不同的业务分支... }`这类逻辑判断；主题差异只能通过Design Token和可替换的动画/图标资源实现。
4. 新增视觉方向或对比模式不需要修改Core、Repository或IPC层代码，只需新增Token集和UI资源。
5. Theme B的对比模式覆盖范围可以少于Theme A（见M4-05实施内容第9条），但不得共用同一套Token导致两个主题在浅色模式下无法区分。

## 结果

### 正面

- 未来增加第三种视觉方向时，只涉及UI层改动，不影响已验证的Core业务逻辑。
- 候选采纳等P0安全路径的测试只需覆盖一套逻辑，不必因主题数量成倍增加测试矩阵。

### 代价

- Theme B的动画（盖章）需要在设计阶段就约束在"纯表现层可替换"的范围内，不能设计成依赖额外业务状态的动画。
- 视觉团队和工程团队需要在Token命名规范上保持纪律，防止个别组件为了视觉效果偷偷加主题判断。

## 强制约束

- 代码审查中发现的主题相关`if/switch`业务分支视为违反本ADR，须拒绝合并。
- `candidate.apply`、`version.create`等P0命令的自动化测试不区分主题运行，且必须全部通过。

## 验证

- 切换主题前后，对同一组Candidate执行采纳操作，产生的Patch、Revision和ApplyRecord必须逐字节一致。
- 静态代码扫描：组件目录中不出现主题字符串与业务命令名同时出现在同一条件分支的模式。
