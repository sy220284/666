# WorldForge 功能清单与实现关系说明

> 状态：Approved  
> 基线：WorldForge V6.5  
> 目的：将完整方案转成可开发、可测试、可追踪的功能明细。每项说明设计方式、需要达到的效果、建议实现和与其他功能的关系。

## 1. 功能关系总图

```text
项目与工作空间
├─ 规划：任务书 → 大纲 → 章节 → 场景
├─ 设定：人物/地点/势力/道具/能力/规则
├─ 正文：Draft → 编辑事务 → Version
├─ AI：约束包 → GenerationRun → Candidate → 比较/采用
├─ 连续性：Canon + EntityState + 时间线 + 知情 + 伏笔 + 尾快照
├─ 质量：校验 + 搜索 + 词典 + 修订待办
└─ 安全交付：回收站 + 备份/恢复 + 导入/导出
```

关键关系：

1. `project.sqlite`是所有项目功能的数据真源。
2. 规划与设定为AI约束包提供输入，但不得自动重写正文。
3. AI输出先形成Candidate，作者采用后才通过事务进入Draft。
4. Draft定稿形成不可变Version；状态提案经作者确认后更新连续性。
5. 搜索、摘要、日记、校验和FTS索引均为派生能力，不覆盖权威数据。

| TIM-001 | 时间线 | 事件记录起止、精度、人物、地点和前置事件 | 发现顺序、持续和同地冲突 | TimelineEvent + dependency关系 | 与章节、状态、校验和检索关联 |
| KNO-001 | 知情信息 | 记录人物知道、相信、怀疑或误解的内容 | 对话和悬疑不泄露未得知信息 | KnowledgeState表 + 来源锚点 | 进入约束包；信息泄露校验使用 |
| FSH-001 | 伏笔 | 埋设、强化、部分揭示、揭示和取消五阶段 | 长线承诺可追踪，超期可提醒 | Foreshadowing + chapter links + dependency关系 | 场景绑定、约束包、校验和日记使用 |
| SNP-001 | 章节尾快照 | 定稿后保存下一章需要的最小状态入口 | 下一章无需重新扫描全部历史 | EndingSnapshot保存实体状态引用和未完成悬念 | 约束包优先读取；返修后可能标记stale |
| STA-002 | 状态提案 | AI只输出旧值、新值、证据和置信等级 | 作者确认后才改变权威状态 | StateProposal状态机：pending/accepted/edited/rejected | 定稿流程触发；可生成项目日记候选 |
| ARC-001 | 人物弧光定义 | 作者为人物创建弧光（成长/黑化/觉醒/堕落/救赎等类型），设定起止意图 | 长线人物转变有明确追踪对象，不再散落在Canon里 | CharacterArc表：entity_id、arc_type、status | 与EntityState、时间线同层，供状态提取和校验引用 |
| ARC-002 | 弧光节点与状态机 | 弧光拆分为里程碑节点，各节点绑定章节，状态为planned/hit/skipped | 转折点可追踪是否按计划推进，而不是全靠回忆 | ArcMilestone表：arc_id、chapter_id、milestone_type、status、depends_on | 状态提取Run识别到疑似命中时生成StateProposal，作者确认后推进status（见ADR-006） |
| ARC-003 | 弧光一致性校验 | 检测当前EntityState性格/行为标签与所处弧光阶段是否矛盾 | 避免人物"无预警"性格反差 | 读取CharacterArc当前阶段与最近EntityState对比，产出ValidationIssue | 并入VAL-003语义校验管线 |
| ARC-004 | 弧光时间线关联 | 弧光节点可依赖或先于特定TimelineEvent | 转折点和剧情事件顺序一致，不会脱节 | ArcMilestone.depends_on引用TimelineEvent | 时间线校验联动 |

## 5. 正文编辑、版本与锁定

| ID | 功能 | 如何设计 | 实现效果 | 如何实现 | 关系与交互 |
|---|---|---|---|---|---|
| EDT-001 | 块级正文 | 段落、对话、标题、分隔符作为DraftBlock | 支持来源、锁定、差异和局部采用 | Tiptap节点与DraftBlock映射；`logicalBlockId`稳定 | Candidate、Version、批量替换共用 |
| EDT-002 | 自动保存 | 800ms空闲或明确事务提交，不逐按键创建Version | 输入流畅且异常退出损失最小 | Renderer收集Patch，Core单写队列事务提交 | Revision递增；顶部显示保存状态 |
| EDT-003 | 中文输入法安全 | composition期间不发送破坏性Patch | 拼音、五笔等输入不丢字、不重复 | 监听compositionstart/end并合并事务 | 自动保存、撤销和Diff均需测试 |
| EDT-004 | 粘贴清理 | 清除网页字体、颜色、脚本和无关样式 | 复制资料不会污染正文格式 | Tiptap paste rule和白名单Schema | 导入功能使用独立解析器 |
| EDT-005 | 锁定块 | UI和Core双层保护，锁定状态低干扰显示 | AI、替换和结构操作不能碰关键段落 | Editor Lock extension + Core LockGuard | 所有Patch、Candidate采用和批量操作前校验 |
| EDT-006 | 撤销重做 | 编辑事务和Candidate采用均可整体撤销 | 作者可大胆尝试且理解回退边界 | 编辑器Undo + 持久化ApplyRecord/Checkpoint | 重启后通过采用前检查点恢复 |
| VER-001 | 不可变Version | 保存版本、定稿和历史恢复都创建新记录 | 历史文本永不被覆盖 | Version和VersionBlock无业务UPDATE路径 | 导出只选择Version；恢复创建新Draft |
| VER-002 | Revision冲突 | AI基于旧Revision生成时仍保存Candidate，采用时逐块检查 | 作者编辑期间AI不串稿、不静默覆盖 | baseDraftRevision + expectedHash + logicalBlockId Diff | 触发冲突界面或安全自动合并 |

## 6. AI接入、约束包与生成

| ID | 功能 | 如何设计 | 实现效果 | 如何实现 | 关系与交互 |
|---|---|---|---|---|---|
| AI-001 | Provider配置 | 统一保存协议、地址、模型、超时和credentialRef | 同时连接外部API和用户已运行的本地服务 | ProviderConfig；凭据放系统Credential Store | 设置中心、连接测试、GenerationRun使用 |
| AI-002 | 连接测试 | 只验证端点可达、认证和最小生成能力 | 配置错误可在生成前发现 | Core直接请求Provider，返回结构化诊断 | 不记录密钥和完整响应正文 |
| AI-003 | 约束包 | P0代码约束、P1章节必须项、P2设定状态、P3文风、P4背景分层 | 有限上下文优先保留真正重要的信息 | 结构关联→尾快照→FTS5→时序过滤→去重→Token裁剪 | T0、T1、局部改写和校验共用 |
| AI-004 | T0骨架 | 生成多个结构化场景方案，不直接写长正文 | 低成本比较剧情方向 | Prompt版本 + JSON Schema + Candidate(type=skeleton) | 选定骨架进入T1；可绕过 |
| AI-005 | T1扩写 | 基于选定骨架和约束包生成完整候选 | AI不会替代当前稿，作者可比较修改 | 流式GenerationRun，完成后一次保存Candidate | 候选审阅、校验、取消和部分结果关联 |
| AI-006 | 快速改写 | 单段内联预览，应用后可立即撤销 | 高频小改不打开完整审阅页 | 选区+邻段+最小约束；返回局部Patch Candidate | 超过范围自动升级为结构性改写 |
| AI-007 | 结构性改写 | 跨段、跨场景和整章进入完整候选流程 | 大改动保持可审计和可比较 | 完整Candidate + baseRevision + Diff | 与锁定、冲突、版本和校验联动 |
| AI-008 | 融合 | 按节拍选择不同候选并只生成必要过渡 | 作者组合优势段落，不必整章重写 | BeatSourceMapping + TransitionStitcher | 生成新的merge Candidate |
| AI-009 | 运行状态 | 展示真实阶段、时长、已接收文本、取消和失败 | 长请求期间作者知道系统在做什么 | MessagePort事件：started/stage/delta/completed/failed/cancelled | 切章不取消，任务条全局可见 |
| AI-010 | 模型支持档案 | 按Provider、模型和任务记录已验证/有限/未验证 | 不把所有模型宣称为同等可靠 | 本地Eval报告和ModelSupportProfile | 决定T0/T1、状态提取和连续性功能提示 |

## 12. 功能实现共性要求

每个功能必须同时定义：

1. 正常路径、失败路径、取消路径和冲突路径。
2. 输入输出Schema、权限范围和项目边界。
3. 是否修改权威数据；若修改，必须明确事务和Revision。
4. 是否受锁定块约束。
5. 是否需要恢复点。
6. 用户界面的空、加载、成功、失败和恢复状态。
7. 单元、集成、E2E、安全或性能测试。

本清单中的功能ID是追踪矩阵、任务卡、测试用例和代码注释引用的统一编号。
