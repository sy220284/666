# WorldForge V1.0任务体系重排基线

> 状态：Frozen  
> 日期：2026-07-14  
> 依据：作者明确要求根据V6.5全功能依赖重新设计任务卡。  
> 适用：仅任务阶段、任务编号、依赖、阶段门和执行顺序。

## 1. 取代范围

本文件与以下文档共同构成新的V1.0任务执行基线：

- `docs/tasks/TASK_INDEX.md`
- `docs/roadmap/V1.0_ROADMAP.md`
- `docs/product/V1.0_TRACEABILITY_MATRIX.md`
- `docs/tasks/M0_TASKS.md`至`M8_TASKS.md`

它们取代以下历史任务路线内容：

- `docs/product/WORLDFORGE_V6.5_FULL_SPEC.md`第10章“任务路线图”中的M0—M5/M0.5划分；
- 旧版M0—M5共27张任务卡；
- 旧版README与开发指南中的24/27任务统计；
- 任何仍以旧任务编号表达执行顺序的说明。

完整产品功能、架构、安全、数据、UI和验收语义仍以V6.5完整规格及专项冻结文档为准。

## 2. 重排原因

旧任务体系存在：

1. 缺少无AI基础写作产品门；
2. Volume/Chapter晚于Draft编辑器；
3. SceneBeat在模型建立前被结构任务使用；
4. 恢复点晚于拆并章、导入和批量替换；
5. FTS在AI约束包和用户搜索中重复规划；
6. app.sqlite、Core生命周期和测试基建没有独立任务；
7. 人物弧光、网文节奏和双视觉主题被塞入旧任务，容量与验收不清；
8. M1—M4用户功能可能到“完整UI”阶段才第一次可操作。

## 3. 新任务阶段

```text
M0 工程、安全与运行底座
→ M1 基础写作MVP
→ M2 编辑安全与版本核心
→ M3 规划、设定与连续性
→ M4 检索与AI基础设施
→ M5 AI生成与候选审阅
→ M6 校验、搜索与交付
→ M7 完整UI与体验整合
→ M8 发布硬化与验收
```

共48张独立任务卡。

## 4. 基础产品门

M1完成时，即使未配置AI，也必须能够：

```text
创建/打开项目
→ 建卷建章
→ 中文写作
→ 自动保存
→ 字数与当前章查找
→ 手动Version与定稿
→ TXT/Markdown导入导出
→ 数据损坏只读与恢复副本
```

M1未Verified前，AI、人物弧光、节奏分析和完整主题骨架不得计入主线完成度。

## 5. 执行规则

- `ACTIVE_TASK.md`是唯一活动任务控制文件。
- 任务只能引用已经Verified的上游表、契约、模型和恢复能力。
- 每张用户功能任务必须包含最小可操作UI。
- M7负责统一整合，不负责第一次接通业务。
- `work`分支不受本次任务文档重排影响；后续需按新任务卡另行评估和拆分。
