# WorldForge V1.0任务体系重排基线

> 状态：Frozen  
> 日期：2026-07-14  
> 增补日期：2026-07-24  
> 依据：作者明确要求根据V6.5全功能依赖重新设计任务卡，并在M5前插入产品体验收口任务。  
> 适用：仅任务阶段、任务编号、依赖、阶段门和执行顺序。

## 1. 当前任务执行基线

本文件与以下文档共同构成V1.0任务执行基线：

- `docs/tasks/TASK_INDEX.md`
- `docs/roadmap/V1.0_ROADMAP.md`
- `docs/product/V1.0_TRACEABILITY_MATRIX.md`
- `docs/tasks/M0_TASKS.md`至`M8_TASKS.md`

历史版本曾使用M0—M5/M0.5和24/27张任务卡。2026-07-14重排形成M0—M8九阶段、52张任务卡；2026-07-24根据已完成部分的产品使用逻辑与用户体验审计，增补`M5-00 作者工作流与产品体验收口`，当前任务体系为九阶段、53张任务卡。

完整产品原则与功能边界以`WORLDFORGE_V6.5_FULL_SPEC.md`为准；具体范围、功能ID、Schema、IPC、AI、UI和P0验收以完整规格列出的专项唯一真源为准。

## 2. 重排原因

旧任务体系存在：

1. 缺少无AI基础写作产品门。
2. Volume/Chapter晚于Draft编辑器。
3. SceneBeat在模型建立前被结构任务使用。
4. 恢复点晚于拆并章、导入和批量替换。
5. FTS在AI约束包和用户搜索中重复规划。
6. app.sqlite、Core生命周期和测试基建没有独立任务。
7. 人物弧光、网文节奏和双视觉主题容量与验收不清。
8. 用户功能可能到完整UI阶段才第一次可操作。
9. Renderer实际实现偏离冻结的React/Zustand架构，若拖到M7将与完整体验整合形成高风险双重重写。
10. M0—M3完成后，底层能力已可靠但界面仍存在工程术语侵入、操作层级过密、继续写作缺失、设定编辑工程化和高风险操作依赖`prompt/confirm`等体验债务；若直接进入M5会放大AI交互复杂度。

## 3. 当前九阶段

```text
M0 工程、安全与运行底座
→ M1 基础写作MVP
→ M2 编辑安全与版本核心
→ M3 规划、设定与连续性
→ M4 检索与AI基础设施
→ M5 作者体验前置、AI生成与候选审阅
→ M6 校验、搜索与交付
→ M7 完整UI与体验整合
→ M8 发布硬化与验收
```

共53张独立任务卡。

M5阶段先执行`M5-00`。该任务依赖M4阶段Verified，且是`M5-01`的硬依赖；M5-00未Verified前不得新增面向用户的T0/T1生成入口。

## 4. 基础产品门

M1完成时，即使未配置AI，也必须能够：

```text
创建/打开项目
→ 建卷建章
→ 中文写作
→ 自动保存
→ 字数与当前章查找
→ 手动Version与定稿
→ TXT/Markdown导入导出
→ 数据损坏只读与恢复副本
```

M1未Verified前，AI、人物弧光、节奏分析和完整主题骨架不得计入主线完成度。

## 5. 依赖与执行规则

- `ACTIVE_TASK.md`是唯一活动任务控制文件。
- 具体任务依赖必须对应已Verified任务。
- 阶段依赖如`M3`表示该阶段全部任务Verified。
- 任务只能引用已经建立的上游表、契约、模型和恢复能力。
- 每张用户功能任务必须包含最小可操作UI。
- Planned任务激活时必须补齐`TASK_TEMPLATE.md`规定的执行附件。
- M3-07—M3-10负责在M4前完成Renderer架构迁移；M5-00负责在AI生成功能前完成现有作者工作流体验收口；M7负责统一整合，不负责第一次接通业务或基础框架重写。
- `work`分支不受本次文档重排和统一修改影响；其代码需按新任务体系另行评估和拆分。
