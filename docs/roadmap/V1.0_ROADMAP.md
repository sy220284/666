# WorldForge V1.0 开发路线图

> 状态：Frozen  
> 基线：WorldForge V6.5  
> 原则：先安全运行，再交付无AI基础写作产品；高级能力只能建立在已验证基础上。

## 1. 总体顺序

```text
M0 工程、安全与运行底座
→ M1 基础写作MVP
→ M2 编辑安全与版本核心
→ M3 规划、设定与连续性
→ M4 检索与AI基础设施
→ M5 AI生成与候选审阅
→ M6 校验、搜索与交付
→ M7 完整UI与体验整合
→ M8 发布硬化与验收
→ V1.0
→ 真实作者使用与长篇数据收集
→ V1.5单独立项
```

## 2. 为什么重排

旧路线在M1之前没有明确的基础可用产品门，且存在章节模型、恢复点、FTS和SceneBeat等依赖倒置。新路线做出四项修正：

1. M1明确交付无AI也能长期写作的基础产品。
2. Volume/Chapter、app.sqlite、Core生命周期和恢复点前置。
3. FTS、Candidate、备份和Prompt拆为公共底座与上层使用，避免重复实现。
4. 每个功能任务同时接通最小UI，M7只做统一整合。

## 4. 关键阶段门

### M0退出门

- 干净环境可安装、lint、typecheck、test、build。
- Electron安全壳、Core监管、SQLite、IPC和测试基建可运行。
- 显示/DPI、AI输出模式和中文Diff有量化决策。

### M1基础产品门

作者必须能够在不配置AI的情况下：

```text
创建/打开项目
→ 建卷建章
→ 中文写作
→ 自动保存
→ 字数与当前章查找
→ 手动版本与定稿
→ TXT/Markdown导入导出
→ 数据损坏只读与恢复副本
```

M1未Verified前，不得把后期AI、人物弧光、节奏检测或主题代码计为主线完成度。

### M2数据安全门

- 所有正文写入统一经过Block Patch、Revision、Hash与LockGuard。
- Candidate隔离、冲突、采用和撤销在Fixture场景中先验证。
- 拆章、并章和永久删除均有恢复点。

### M3权威连续性门

- 规划不自动修改正文。
- AI或规则产生的状态变化只进入StateProposal。
- pending提案不改变EntityState或ArcMilestone。

### M4 AI基础门

- FTS和约束包可追溯。
- Provider凭据不进入数据库和Renderer。
- Prompt、输出Schema、Cleaner和Eval版本化。
- AI不可用不影响M1—M3。

### M5作者可控AI门

- AI输出始终先成为Candidate。
- T0可绕过，T1支持纯文本流降级。
- 取消、断流和partial结果不改变Draft。

### M6交付门

- 校验和节奏只提示，不替作者裁决。
- 全项目替换、DOCX和备份恢复覆盖失败与恢复路径。

### M7体验门

- 所有已实现功能有真实入口。
- 两种视觉方向不分叉业务逻辑。
- 1280×800、2K、21:9和混合DPI可完成核心流程。

### M8发布门

- P0-001—P0-075全部Verified或明确Blocked。
- 安全、数据、性能、E2E、AI Eval和跨平台证据完整。
- README和用户文档只描述真实可用功能。

## 5. 并行规则

- M0-02与M0-03可在M0-01后并行，但M0-04必须等待两者。
- M1-08可与M1-03—M1-07部分并行，M1-09必须等待版本和恢复点。
- 同一阶段内仅依赖已满足且不修改同一权威契约的任务可并行。
- 不得并行修改同一Migration序列、IPC命令、Draft/Candidate/Version模型或Prompt版本。

## 6. V1.5启动门

V1.5不得按日期自动开始，必须满足：

1. V1.0已被真实作者稳定使用。
2. 已有中型或长篇项目数据。
3. FTS召回不足、失效传播成本或自动日记价值有可重复证据。
4. Core性能仍有余量或已达到明确拆分阈值。

V1.5包括L0—L5自动分层记忆、卷级检查点、定时项目日记、超长篇专项适配及有证据时的语义检索。
