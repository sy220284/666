# WorldForge V6.5 最终工程设计方案（第2部分）

# 5. 本地工作空间与数据权威

## 5.1 目录结构

```text
WorldForgeData/
├── app.sqlite # 最近项目、全局设置、Provider元数据
└── workspaces/
    └── <project>.worldforge/
        ├── manifest.json
        ├── project.sqlite # 项目唯一权威数据源
        ├── attachments/
        ├── exports/
        ├── backups/
        └── logs/
```

## 5.2 数据权威规则

- 正文、人物、设定、时间线、伏笔、候选、版本和状态全部写入project.sqlite。
- 编辑器文档、缓存和导出文件不是权威数据源，可以重新生成。
- app.sqlite只保存应用级配置和项目路径，不保存作品正文。
- 密钥只保存在操作系统Credential Store，数据库仅保存credentialRef。

## 5.3 事务边界

| **操作** | **事务要求** |
|---|---|
| 保存Draft Patch | 校验Revision与锁定块后一次提交 |
| 接受Candidate | 冲突检查、写入Draft、来源记录、Revision递增一次提交 |
| 创建Version | 复制Draft快照、创建父关系、更新章节指针一次提交 |
| 确认状态提案 | 更新EntityState、生成尾快照一次提交 |
| 迁移 | 每个Migration独立事务，失败保持旧Schema可用 |

# 6. 正文编辑、Draft、Candidate与Version

> **图示说明：** 正文、Draft、Candidate与Version关系图。

## 6.1 块级正文模型

```ts
interface DraftBlock {
  id: string; // 当前Draft记录ID
  draftId: string;
  logicalBlockId: string; // 跨候选和版本追踪的逻辑身份
  orderKey: string;
  type: 'paragraph' | 'dialogue' | 'heading' | 'separator';
  text: string;
  source: 'manual' | 'ai' | 'mixed' | 'imported';
  locked: boolean;
  contentHash: string;
  revision: number;
}
```

## 6.2 Draft

- 每章只有一个活动Draft，作为唯一可编辑正文。
- 一次事务无论修改多少块，只将Draft Revision增加1。
- AI运行记录baseDraftRevision；接受结果时必须与当前Revision比较。

## 6.3 Candidate

- AI输出保存为候选，不自动写入Draft。
- 候选可以是骨架、完整章节、局部重写或融合稿。
- 作者可以接受全部、选择局部、手动融合或丢弃。
- 流式中断的部分结果可以保存为partial，但不得直接定稿。

## 6.4 不可变Version

- Version创建后不得修改或覆盖。
- 恢复历史版本的方式是创建新的Draft，不修改旧Version。
- 定稿Version可以继续被引用、导出和比较。

## 6.5 锁定与冲突

| **场景** | **处理** |
|---|---|
| AI重写包含锁定块 | Core拒绝任务或自动排除锁定块 |
| 用户在AI生成期间编辑Draft | 候选仍可保存；接受时进行Revision冲突检测 |
| 候选块Hash与当前块不一致 | 进入逐块冲突处理，不静默覆盖 |
| 历史版本块ID不同 | 使用logicalBlockId追踪逻辑身份 |

# 7. 创作规划与长篇连续性

## 7.1 作品任务书

作品任务书是全书最高层约束，但保持简洁。建议字段包括：一句话高概念、核心阅读承诺、主角目标、核心冲突、终局方向、必须兑现内容、明确禁止内容和目标篇幅。

## 7.2 大纲与节拍

| **实体** | **核心字段** |
|---|---|
| PlotNode | 层级、标题、目标、核心冲突、预期结果、状态 |
| Chapter | 卷、顺序、标题、Draft、Final Version、状态 |
| SceneBeat | 目标、冲突、结果、人物、地点、预计字数比例、是否必选 |

## 7.3 人物与世界设定

- 人物：目标、欲望、恐惧、行为边界、关系、声音样本。
- 世界：规则、地点、势力、道具、能力和关键事件。
- 静态事实与动态状态分离；AI只能对动态状态提出变更候选。

## 7.4 时间线

v1实现简单时间线，不开发完整历法引擎。事件包含开始、结束、精度、人物、地点和前置事件。校验重点是顺序冲突、人物同一时间出现在不同地点、持续时间明显不合理。

## 7.5 知情信息

系统记录重要信息、秘密或误解由哪些人物知道、何时得知、是否相信。该信息用于悬疑、对话潜台词和行为合理性检查，不实现复杂传播模拟。

## 7.6 伏笔

伏笔保留埋设、强化、部分揭示、揭示和取消五种主要状态，并记录关联章节、建议回收窗口和依赖。v1不实现复杂图算法，只用关系表保存必要依赖。超长篇项目中的分层记忆、时序状态和检查点规则见第21章。

# 8. AI接入、约束包与生成工作流

## 8.1 Provider配置

```ts
interface ProviderConfig {
  id: string;
  name: string;
  protocol: 'openai_compatible' | 'anthropic' | 'custom';
  baseUrl: string;
  model: string;
  credentialRef?: string;
  timeoutMs: number;
  advancedOptions?: Record<string, unknown>;
}
```

本地模型服务与外部API使用同一配置方式。WorldForge只连接已经运行的端点，不负责模型下载、启动、容器和显存管理。

## 8.2 最小能力信息

```ts
interface ModelCapabilities {
  streaming: boolean;
  structuredOutput: boolean;
  maxContextTokens: number;
  maxOutputTokens: number;
}
```

## 8.3 GenerationRun

```ts
interface GenerationRun {
  id: string;
  requestId: string;
  chapterId: string;
  runType: 'skeleton' | 'chapter' | 'rewrite' | 'merge' | 'validate' | 'state_extract';
  baseDraftRevision: number;
  providerId: string;
  model: string;
  status: 'queued' | 'running' | 'succeeded' | 'failed' | 'cancelled';
  retryCount: number;
  errorCode?: string;
}
```

- 每次用户点击生成创建一个新的requestId。
- 同一requestId重复提交不重复执行，用于防止按钮连点。
- 用户主动重新生成会创建新的Run，不使用generationFingerprint或Nonce。
- 网络失败在同一个Run上记录retryCount。

## 8.4 约束包

| **优先级** | **内容** |
|---|---|
| P0 程序硬约束 | 锁定块、目标项目、目标章节、不可变Version、Revision |
| P1 本章必须满足 | 核心事件、必选节拍、前章尾状态、必须处理的伏笔 |
| P2 高相关设定 | 出场人物、当前地点、关键道具、世界规则 |
| P3 表现要求 | 作品规则、角色声音、场景节奏、作者样本 |
| P4 补充背景 | 低相关历史和远期资料；Token不足时优先删除 |

## 8.5 检索

基础检索使用结构化关联和SQLite FTS5：先读取与章节明确关联的人物、地点、道具、伏笔和当前状态，再用名称、别名和关键词从设定与正文中补充召回。超长篇项目必须追加时序有效性过滤、卷/剧情弧记忆和热温冷分区；语义检索仅在百万字评测证明FTS5召回不足后立项，不预建通用向量平台。

> **YAGNI约束：** v1不建立向量检索接口、Embedding表或Rerank模型。只有真实项目评测证明FTS5长期不足时，才重新设计。

## 8.6 T0/T1与局部重写

| **任务** | **输入** | **输出** |
|---|---|---|
| T0骨架 | 约束包、节拍、倾向 | 多个结构化骨架Candidate |
| T1正文 | 约束包、选定骨架、文风 | 完整章节Candidate |
| 局部重写 | 目标块、指令、上下文、expectedHash | 局部Patch Candidate |
| 融合 | 节拍来源选择、基础版本 | 融合Candidate与过渡修补 |

## 8.7 Candidate接受

- 比较候选的baseDraftRevision和当前Draft Revision。
- 校验目标块是否仍存在、Hash是否一致、是否新增锁定。
- 无冲突时一次事务写入Draft并增加Revision。
- 存在冲突时展示当前稿、候选稿和手动合并入口。
