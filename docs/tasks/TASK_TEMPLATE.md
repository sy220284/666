# WorldForge 任务卡模板

> 状态：Frozen  
> 适用：`docs/tasks/M0/`至`M8/`  
> 原则：计划卡保持可导航；任务激活时补齐可执行细节，避免48张未启动任务重复维护易漂移内容。

## 1. 两层任务结构

### 1.1 Planned任务卡

仓库中的独立任务卡在`Planned`阶段必须包含：

1. 基本信息：任务ID、状态、里程碑、优先级、建议分支。
2. 目标。
3. 阶段定位。
4. 非目标。
5. 依赖。
6. 关联需求、功能ID和P0验收。
7. 必读文档。
8. 主要影响范围。
9. 实施内容。
10. 测试与证据。
11. 完成条件。

现有48张任务卡使用该结构，不要求在未激活时复制全部Schema、IPC和失败表格。

### 1.2 Active执行附件

任务被写入`ACTIVE_TASK.md`时，必须在活动任务控制文件中补齐：

- 精确前置任务和已Verified证据。
- 输入与输出。
- 数据库表、字段、Migration、事务与回滚。
- IPC命令、事件、错误码和Preload白名单。
- 最小UI闭环。
- 安全与隐私。
- 失败、取消、冲突、只读和恢复路径。
- 自动化测试、人工验收和性能预算。
- 证据目录和完成报告格式。

缺少执行附件时，任务不得进入`In Progress`。

## 2. 依赖写法

依赖可以使用两种形式：

- 具体任务：`M1-04、M1-05`，表示这些任务必须Verified。
- 阶段门：`M3`，表示M3全部任务必须Verified。

禁止使用含义不明的“前期任务”“相关基础”或未列入任务索引的旧编号。

## 5. Definition of Done

任务只有同时满足以下条件才能标记Verified：

- 目标真实接通，非目标未提前引入。
- 上游任务和阶段门已Verified。
- 用户功能有最小可操作UI。
- 成功、失败、取消、冲突、只读和恢复路径已覆盖。
- Schema、Migration、IPC、事件、错误码、UI、安全和文档同步。
- 测试真实运行并记录退出状态。
- 证据保存到`docs/test-evidence/<TASK-ID>/`。
- 无TODO、空函数、固定假数据和伪造成功。
- `TASK_INDEX.md`和追踪矩阵已更新。

## 6. 完成报告

记录：变更摘要、文件、数据库/IPC变化、测试命令、人工验收、性能、已知限制、剩余风险和下一候选任务。任务关闭后不得自动开始下一任务。
