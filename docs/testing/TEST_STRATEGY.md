# WorldForge 测试策略

> 状态：Frozen  
> 目标：证明功能真实可用、数据边界未被破坏，并为每个“完成”结论提供可复核证据。

## 1. 测试原则

1. 代码硬保证与模型质量分开验收。
2. 高风险功能先写失败测试或稳定复现，再实现。
3. 业务写入测试必须覆盖成功、失败、取消和冲突。
4. 数据库、IPC和UI不能只用Mock互相证明；关键闭环必须真实联通。
5. 测试Fixture不得包含用户私人作品内容。
6. 未运行的测试不得在完成报告中写成通过。

## 2. 测试层级

| 层级 | 工具 | 覆盖 |
|---|---|---|
| 单元测试 | Vitest | 领域不变量、Patch、锁定、Diff、Token裁剪、错误映射 |
| Repository测试 | Vitest + 临时SQLite | SQL、索引、事务、外键、并发和数据归属 |
| 集成测试 | Vitest | Renderer契约→Preload→Core Use Case→SQLite/Provider Stub |
| Migration测试 | Vitest + Schema Fixtures | 新库、旧库升级、中断、重复执行和兼容处理 |
| 安全测试 | Vitest/Playwright | Electron隔离、IPC白名单、路径边界、日志脱敏和导入限制 |
| 桌面E2E | Playwright Electron | 真实窗口中的核心创作流程、错误和恢复路径 |
| 性能测试 | 自定义基准 + Playwright | 键入、自动保存、IPC、Diff、FTS、打开项目和高分屏滚动 |
| AI Eval | 固定Fixture + Provider适配器 | T0/T1、改写、状态提取、连续性和禁止信息泄露 |
| 手动验收 | 任务脚本 | 视觉、输入法、2K/曲面、系统对话框和真实Provider |

## 3. 必须保持为0的代码缺陷

- 锁定块被修改。
- 未确认Candidate写入Draft。
- Revision冲突静默覆盖。
- AI直接修改Canon或权威状态。
- 跨项目读写。
- 凭据进入普通日志或数据库。
- 恢复覆盖原项目。

这些指标不因模型能力或时间压力降低。

## 4. 单元测试重点

### Domain

- 实体、状态、伏笔和章节状态迁移。
- `logicalBlockId`规则。
- 不可变Version。
- 约束优先级和裁剪。

### Editor Core

- Patch插入、更新、删除和移动。
- `expectedHash`冲突。
- 锁定块拒绝。
- 中文IME事务合并。
- 粘贴清理。
- 字数统计口径。

### Prompt与AI解析

- T0 Schema。
- 状态提案Schema。
- 模型代码块和多余前后缀清理。
- 无效输出明确失败，不无限猜测。

## 5. 数据库与集成测试

每个写Use Case测试：

1. 正常提交。
2. 参数不合法。
3. 目标不存在。
4. 项目ID不匹配。
5. Revision或Hash冲突。
6. 锁定冲突。
7. 事务中途故障注入。
8. 重复`requestId`。
9. 关闭和重启后状态一致。

自动保存与AI并发测试至少执行100轮，不允许`SQLITE_BUSY`直接泄露给用户。

## 6. Migration测试

- 空库升级到最新。
- 每个已支持旧版本逐级升级。
- 迁移中断后重启。
- checksum不一致。
- 高版本数据库只读打开。
- 大型数据回填的恢复。
- 迁移后`foreign_key_check`、`quick_check`和核心业务查询。

## 7. E2E核心场景

### E2E-01 自主写作

新建项目→创建章节→输入中文→自动保存→锁定段落→关闭重开→正文与锁定状态一致。

### E2E-02 AI候选

配置Stub→T0→选骨架→T1→切换章节→返回→比较→采用部分段落→撤销→重新采用→定稿Version。

### E2E-03 Revision冲突

启动生成→作者修改同一段→候选完成→采用→显示冲突→保留当前或采用候选→无静默覆盖。

### E2E-04 连续性

创建人物和状态→定稿→生成状态提案→接受→下一章约束包包含当前状态→返修前章→后续快照标记stale。

### E2E-05 导入导出

导入TXT/DOCX预览→调整分章→提交→定稿→导出选定Version→内容一致。

### E2E-06 备份恢复

创建日常和手动备份→修改正文→恢复到新目录→原项目不变→恢复副本通过检查。

## 8. 安全测试

- Renderer无法访问Node、文件、数据库和环境变量。
- 未注册IPC和额外字段被拒绝。
- `../`、绝对路径和符号链接逃逸被拒绝。
- 跨项目实体ID被拒绝。
- 外链只用系统浏览器打开。
- DOCX异常压缩比、过多文件、宏格式和外部资源安全处理。
- 日志与诊断包自动扫描敏感样本。

## 9. 性能基线

| 指标 | V1目标 |
|---|---:|
| 2K键入P95 | ≤50ms |
| 自动保存事务P95 | ≤150ms |
| 编辑IPC P95 | ≤200ms |
| AI取消反馈 | ≤500ms |
| 5000字Diff首屏 | ≤500ms |
| 5000字完整Diff | ≤1.2s |
| 正文滚动 | ≥50fps |
| Core单次事件循环阻塞 | <100ms |

性能测试报告必须记录机器、系统缩放、数据规模和样本数。

## 10. 显示测试矩阵

- 1280×800，100%。
- 2560×1440，100%、125%、150%。
- 3440×1440。
- 3840×1600。
- 混合DPI双显示器切换。
- 1024×640有效视口。

检查正文清晰度、侧栏折叠、弹层位置、候选比较模式、焦点顺序和无整页横向滚动。

## 11. AI Eval

按`Provider + Model + Task + PromptVersion`保存结果。

任务：

- T0骨架。
- T1扩写。
- 快速改写。
- 状态提取。
- 连续性检查。
- 禁止信息泄露。

模型指标不能替代代码硬保证。未达基线时允许降级或绕过AI流程。

## 13. 完成规则

任务完成报告必须列出真实执行的命令、退出状态、结果摘要、手动步骤和剩余限制。只创建脚手架、Mock或测试文件但未实现功能时，结论必须是“未完成”或“部分完成”。
